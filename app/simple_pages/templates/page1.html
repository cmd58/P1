Git is a free and open-source version control system used to handle small to very large projects.

Here are some key terms to know when thinking about Git:

  • Branch - Branches allow you to develop features, fix bugs, or safely experiment with new ideas in a contained area of your repository.
  • Merge - Git's way of putting a forked history back together again. The git merge command lets you take independent lines of development created by git branch and integrate them into a single branch.
  • Commit - The git commit command captures a snapshot of the project's currently staged changes.
  • Tags - ref's that point to specific points in Git history. Tagging is generally used to capture a point in history that is used for a marked version release (i.e. v1. 0.1).
  • Repositories - contain the collection of files of various different versions of a Project. Files are imported from the repository into the local server of the user for further modifications in the content of the file.

Keeping the definition of these terms in mind, we can begin to understand how to work with Git. The GitHub workflow can be condensed into branching, commits, pull requests, collaboration, and then merge.

The Master Branch

All the code in production is stored in the master branch. Master branch should always be working, so we should create a new branch where we can test out changes and additions.

Push vs Pull

We commit and push on the new branch and then open a pull request. A push forces your commits to the target repo while a pull refers to the target repo grabbing you commits to be present in the repo.

Merging & Forks

If the pull request is approved we can merge the branch and the master. If we don't have write access to the repo we can fork the reop onto our own GitHub account. We can make commits on that fork and then make a pull request on the original fork.
